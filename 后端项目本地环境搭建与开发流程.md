# 后端项目本地环境搭建与开发流程

本文面向第一次使用公司 Windows 电脑参与本项目开发的新员工。完成后，你应能在公司定制的 Ubuntu 24.04 WSL 中使用指定版本的 Java、Maven 和 Node.js，通过 WSL 内的 Docker Engine 启动 MySQL、Redis，在 IDEA 中启动后端、在 VS Code 中启动前端，并按团队分支规范提交 Merge Request（MR）。

本文是一份项目模板。正式放入公司代码仓前，文档维护者必须先替换“公司参数清单”中的全部必填占位符，并用一台新电脑完整走通一次流程。

> **命令环境约定**
>
> - 标有“Windows PowerShell”的命令在 Windows PowerShell 中执行。
> - 标有“WSL”的命令在公司定制的 Ubuntu WSL 终端中执行。
> - 不要把命令提示符 `PS>`、`$` 或示例输出一起复制。
> - 命令中的 `{{...}}` 是待替换占位符，不能原样执行。

## 四、让 WSL 使用 Windows 代理

以下方式适用于 Windows 11 22H2 及以上版本。公司电脑若由 IT 统一管理 `.wslconfig`，应以 IT 下发配置为准。

### 4.1 启用 autoProxy

在 Windows PowerShell 中打开用户级 WSL 配置：

~~~powershell
notepad "$env:USERPROFILE\.wslconfig"
~~~

如果文件已有其他配置，应合并下面的设置，不要覆盖原内容：

~~~ini
[wsl2]
autoProxy=true
~~~

保存后，在 Windows PowerShell 中关闭所有 WSL 实例：

~~~powershell
wsl --shutdown
~~~

重新进入公司发行版：

~~~powershell
wsl -d "{{WSL_DISTRO_NAME}}"
~~~

### 4.2 验证代理

在 WSL 中执行：

~~~bash
env | cut -d= -f1 | grep -iE '^(http|https|no)_proxy$' || true
curl -I https://ubuntu.com/
curl -I https://maven.apache.org/
~~~

如果公司只允许访问内部制品平台，应将测试地址替换为公司批准的内部地址。

浏览器能访问网页不代表 WSL、APT、Maven 或 Docker daemon 都已正确使用代理。这些工具的网络路径不同，需要分别验证。

### 4.3 让 `sudo` 安全继承代理变量

`sudo` 本身通常不会造成网络超时。更常见的原因是：WSL `autoProxy` 已把代理变量设置到当前用户的 Shell，但 `sudo` 默认使用经过清理的最小环境，提权后的 APT、curl 等进程没有继承这些变量。

先区分“`sudo` 本身异常”和“提权后的网络命令没有代理”：

~~~bash
sudo -v
time sudo true

env | cut -d= -f1 | grep -iE '^(http|https|no)_proxy$' || true
sudo env | cut -d= -f1 | grep -iE '^(http|https|no)_proxy$' || true
~~~

- 如果 `sudo true` 很快结束、普通用户能看到代理变量，但 `sudo env` 看不到，按下面步骤配置。
- 如果 `sudo true` 本身也长时间卡住，问题不在代理变量，应联系 WSL 维护者检查本机用户名、主机名解析、PAM 或公司认证配置。

#### 推荐方案：只允许代理变量通过 `sudo`

使用 `visudo` 创建独立配置文件：

~~~bash
sudo visudo -f /etc/sudoers.d/wsl-proxy-env
~~~

在编辑器中只添加下面一行并保存：

~~~sudoers
Defaults env_keep += "http_proxy https_proxy no_proxy HTTP_PROXY HTTPS_PROXY NO_PROXY"
~~~

随后检查权限和全部 sudoers 配置：

~~~bash
sudo chmod 0440 /etc/sudoers.d/wsl-proxy-env
sudo visudo -c
~~~

`visudo -c` 必须报告配置解析成功。重新进入 WSL 后验证：

~~~bash
env | cut -d= -f1 | grep -iE '^(http|https|no)_proxy$'
sudo env | cut -d= -f1 | grep -iE '^(http|https|no)_proxy$'
sudo apt update
~~~

这个配置只保存允许继承的变量名，不会把当前代理地址或凭据写进 sudoers。WSL 每次启动后仍由 `autoProxy` 提供实际值，而 `sudo` 继续保留默认的 `env_reset` 安全边界。

如果公司策略禁止开发者修改 sudoers，应由 WSL 脚本维护者统一下发该文件。在等待统一配置时，可对单次可信命令显式保留这六个变量：

~~~bash
sudo --preserve-env=http_proxy,https_proxy,no_proxy,HTTP_PROXY,HTTPS_PROXY,NO_PROXY apt update
~~~

不要把 `sudo -E` 作为常规解决方案，因为它申请保留整个用户环境，而当前需求只需要代理变量。也不要为了绕过证书错误而使用 `curl -k`、关闭 Git TLS 校验或删除证书验证。

> **Docker 代理是独立配置**
>
> WSL 交互终端中的代理变量不会自动成为 systemd 管理的 Docker daemon 配置。如果 `curl` 正常而 `docker pull` 失败，请继续阅读“配置 Docker daemon 代理”，不要反复修改 Shell 变量。

## 五、安装 Java 17

以下命令在 WSL 中执行。

### 5.1 安装 JDK

~~~bash
sudo apt update
apt-cache policy openjdk-17-jdk
sudo apt install -y openjdk-17-jdk
~~~

验证 Java 运行时和编译器：

~~~bash
java -version
javac -version
readlink -f "$(command -v java)"
readlink -f "$(command -v javac)"
~~~

`java` 和 `javac` 的主版本都必须是 17。只安装 JRE 不足以进行后端编译和测试。

### 5.2 配置 `JAVA_HOME`

先计算当前 JDK 根目录：

~~~bash
jdk_home="$(dirname "$(dirname "$(readlink -f "$(command -v javac)")")")"
printf 'detected JAVA_HOME=%s\n' "$jdk_home"
test -x "$jdk_home/bin/java"
test -x "$jdk_home/bin/javac"
~~~

将配置写入 Bash 启动文件。下面的标记可以避免重复追加：

~~~bash
if ! grep -qF '# company-project-java' "$HOME/.bashrc"; then
  printf '%s\n' \
    '# company-project-java' \
    'export JAVA_HOME="$(dirname "$(dirname "$(readlink -f "$(command -v javac)")")")"' \
    'export PATH="$JAVA_HOME/bin:$PATH"' \
    >> "$HOME/.bashrc"
fi

source "$HOME/.bashrc"
~~~

重新验证：

~~~bash
printf 'JAVA_HOME=%s\n' "$JAVA_HOME"
"$JAVA_HOME/bin/java" -version
"$JAVA_HOME/bin/javac" -version
~~~

## 六、安装 Maven 3.9.0

项目固定使用 Maven 3.9.0。该版本位于 Apache 官方历史归档中；不要因为看到更新版本就自行替换项目要求。

### 6.1 下载并校验

在 WSL 中执行：

~~~bash
MAVEN_VERSION=3.9.0
MAVEN_ARCHIVE="apache-maven-${MAVEN_VERSION}-bin.tar.gz"
MAVEN_BASE_URL="https://archive.apache.org/dist/maven/maven-3/${MAVEN_VERSION}/binaries"
maven_work_dir="$(mktemp -d)"

cd "$maven_work_dir"
curl -fLO "${MAVEN_BASE_URL}/${MAVEN_ARCHIVE}"
curl -fLO "${MAVEN_BASE_URL}/${MAVEN_ARCHIVE}.sha512"

expected_sha512="$(tr -d '[:space:]' < "${MAVEN_ARCHIVE}.sha512")"
printf '%s  %s\n' "$expected_sha512" "$MAVEN_ARCHIVE" | sha512sum -c -
~~~

只有看到 `OK` 才能继续。校验失败时删除本次下载文件并重试，不要跳过校验。

### 6.2 安装到 `/opt`

~~~bash
sudo tar -xzf "$MAVEN_ARCHIVE" -C /opt
sudo ln -sfn "/opt/apache-maven-$MAVEN_VERSION" /opt/maven

cd "$HOME"
rm -rf "$maven_work_dir"
~~~

配置 Maven 环境变量：

~~~bash
if ! grep -qF '# company-project-maven-3.9.0' "$HOME/.bashrc"; then
  printf '%s\n' \
    '# company-project-maven-3.9.0' \
    'export MAVEN_HOME=/opt/maven' \
    'export PATH="$MAVEN_HOME/bin:$PATH"' \
    >> "$HOME/.bashrc"
fi

source "$HOME/.bashrc"
hash -r
mvn -version
~~~

`mvn -version` 必须同时显示：

- Apache Maven 3.9.0。
- Java version 17。
- Maven home 为 `/opt/maven` 或其真实目标目录。

### 6.3 安装公司 `settings.xml`

假设文件已通过公司批准的渠道下载到 Windows 下载目录：

~~~bash
settings_source="/mnt/c/Users/{{WINDOWS_USER_NAME}}/Downloads/settings.xml"
test -f "$settings_source"

mkdir -p "$HOME/.m2"
chmod 700 "$HOME/.m2"

if [ -f "$HOME/.m2/settings.xml" ]; then
  cp -p "$HOME/.m2/settings.xml" \
    "$HOME/.m2/settings.xml.backup.$(date +%Y%m%d-%H%M%S)"
fi

install -m 600 "$settings_source" "$HOME/.m2/settings.xml"
stat -c '%A %n' "$HOME/.m2/settings.xml"
~~~

注意：

- 不要用符号链接让 Maven 直接读取 `/mnt/c` 中的配置。
- 不要把 `settings.xml` 复制进项目目录或提交 Git。
- 不要在公开聊天、Issue 或 MR 中粘贴完整 `settings.xml`。
- 遇到 `401`、`403`、`407` 或 `PKIX path building failed` 时，应检查私有仓权限、代理和企业证书，不要关闭 HTTPS 校验。

## 八、在 WSL 中安装 Docker Engine

公司禁止安装 Docker Desktop。本项目只使用公司定制 WSL 内独立运行的 Docker Engine。

### 8.1 确认 systemd

~~~bash
ps -p 1 -o comm=
systemctl --version
~~~

第一条命令必须输出 `systemd`。若不是，返回 WSL 安装验证步骤并联系维护者。

### 8.2 清理冲突包

先查看是否存在冲突包：

~~~bash
dpkg -l | grep -E 'docker|containerd|runc|podman' || true
~~~

新环境如果装有 Ubuntu 自带或旧版冲突包，按 Docker 官方要求移除：

~~~bash
sudo apt remove -y \
  docker.io \
  docker-compose \
  docker-compose-v2 \
  docker-doc \
  podman-docker \
  containerd \
  runc
~~~

APT 提示某些包未安装是正常的。此命令不会自动删除 `/var/lib/docker` 中已有数据；如果这不是全新环境，先确认数据归属再继续。

### 8.3 配置 Docker 官方 APT 仓库

~~~bash
sudo apt update
sudo apt install -y ca-certificates curl
sudo install -m 0755 -d /etc/apt/keyrings

docker_key_file="$(mktemp)"
curl -fsSL https://download.docker.com/linux/ubuntu/gpg -o "$docker_key_file"
sudo install -m 0644 "$docker_key_file" /etc/apt/keyrings/docker.asc
rm -f "$docker_key_file"

sudo tee /etc/apt/sources.list.d/docker.sources >/dev/null <<EOF
Types: deb
URIs: https://download.docker.com/linux/ubuntu
Suites: $(. /etc/os-release && echo "${UBUNTU_CODENAME:-$VERSION_CODENAME}")
Components: stable
Architectures: $(dpkg --print-architecture)
Signed-By: /etc/apt/keyrings/docker.asc
EOF

sudo apt update
apt-cache policy docker-ce
~~~

如果 `sudo apt update` 因代理超时，按前文在 sudoers 中仅允许代理变量通过，再重新执行。

### 8.4 安装并启动 Docker

~~~bash
sudo apt install -y \
  docker-ce \
  docker-ce-cli \
  containerd.io \
  docker-buildx-plugin \
  docker-compose-plugin

sudo systemctl enable --now docker.service
sudo systemctl enable --now containerd.service
sudo systemctl status docker.service --no-pager
~~~

将当前用户加入 Docker 组：

~~~bash
sudo usermod -aG docker "$USER"
id "$USER"
~~~

Docker 组可以控制 Docker daemon，等价于授予接近 root 的权限。只把经过批准的开发账号加入该组，不要执行 `chmod 666 /var/run/docker.sock`。

退出 WSL，在 Windows PowerShell 中执行：

~~~powershell
wsl --shutdown
wsl -d "{{WSL_DISTRO_NAME}}"
~~~

重新进入后验证：

~~~bash
id
systemctl is-active docker.service
docker version
docker compose version
docker run --rm hello-world
~~~

### 8.5 配置 Docker daemon 代理

只有在公司明确要求，或出现“`curl` 能访问但 `docker pull` 失败”时，才执行本节。

先查看 WSL 当前代理信息：

~~~bash
env | grep -iE '^(http|https|no)_proxy=' || true
~~~

Docker daemon 是 systemd 服务，不会自动继承当前终端变量。根据公司提供的实际代理值，将下面配置合并进 `/etc/docker/daemon.json`：

~~~json
{
  "proxies": {
    "http-proxy": "{{DOCKER_DAEMON_PROXY_URL}}",
    "https-proxy": "{{DOCKER_DAEMON_PROXY_URL}}",
    "no-proxy": "localhost,127.0.0.1,::1,{{DOCKER_NO_PROXY}}"
  }
}
~~~

如果该文件已经存在其他配置，只合并 `proxies` 字段，不要覆盖原内容。代理账号、密码和内部地址必须按公司安全要求处理，不能提交到项目仓。

保存后验证配置并重启：

如果代理使用 PAC、身份认证或 HTTPS 解密证书，应使用 IT 提供的 Docker 专用方案，不要自行把浏览器代理地址硬编码到共享仓库。

## 十、启动 MySQL 和 Redis

以下命令都在代码仓根目录执行：

~~~bash

~~~

### 10.1 为什么显式指定 `.env.local`

Docker Compose 默认自动读取的是项目目录下的 `.env`，不会仅因为文件名是 `.env.local` 就自动将它用于 Compose 插值。因此本文统一显式使用：

~~~text
docker compose --env-file .env.local ...
~~~

不要执行 `docker compose config` 后把完整输出粘贴到聊天或 Issue 中，因为展开后的配置可能包含敏感值。只做语法验证时使用 `--quiet`。

### 10.2 验证并启动

~~~bash
docker compose --env-file .env.local config --quiet

docker compose --env-file .env.local pull \
  "{{MYSQL_SERVICE}}" \
  "{{REDIS_SERVICE}}"

docker compose --env-file .env.local up -d \
  "{{MYSQL_SERVICE}}" \
  "{{REDIS_SERVICE}}"

docker compose --env-file .env.local ps
~~~

查看启动日志：

~~~bash

~~~

如果 Compose 定义了健康检查，应等待 MySQL 和 Redis 显示 `healthy`。MySQL 首次创建数据目录时需要一定时间，在初始化完成前不要立刻导入快照。

### 10.3 验证 MySQL 可接受连接

以下命令从 MySQL 容器已有的本地密码环境变量读取密码，不会把密码写入宿主机命令行：

~~~bash
docker compose --env-file .env.local exec -T "{{MYSQL_SERVICE}}" \
  sh -eu -c '
    MYSQL_PWD="$(printenv "{{MYSQL_PASSWORD_ENV}}")"
    export MYSQL_PWD
    exec mysqladmin --user=root ping
  '
~~~

预期输出 `mysqld is alive`。如果 `{{MYSQL_PASSWORD_ENV}}` 不是容器内真实存在的变量，文档维护者必须在发布前替换为项目实际配置。

> **不要把 `down -v` 当作普通重启**
>
> `docker compose down -v` 会删除当前 Compose 项目的命名卷，可能同时删除 MySQL、Redis 和其他本地数据。日常启停只使用 `stop`、`start`、`restart` 或不带 `-v` 的 `down`。

## 十一、准备和导入脱敏数据库快照

本节分为两个角色：

- 新员工只需执行“新员工：下载并导入快照”。
- 指定的快照维护者才执行“维护者：制作并发布快照”。

任何数据库快照都可能包含业务数据。原始测试库导出、脱敏、保存、传输、保留期限和删除方式必须符合公司数据安全制度。

### 11.1 数据流

~~~text
测试数据库
  ↓ 由授权人员复制到隔离环境
获批的脱敏副本
  ↓ mysqldump 逻辑导出
SQL 文件
  ↓ gzip 压缩 + SHA-256
内部文件或制品平台
  ↓ 新员工下载并校验
WSL 用户目录（项目仓之外）
  ↓ docker compose exec -T
本地 MySQL 容器
  ↓ 关键 SQL + 应用验证
本地功能验证环境
~~~

### 11.2 维护者：脱敏要求

不要直接修改共享测试库来制作快照。应先在获批的隔离副本中完成脱敏，并至少检查：

- 姓名、手机号、邮箱、证件号码和地址。
- 密码摘要、Token、密钥、会话和验证码。
- 支付、账户、订单和其他受保护业务标识。
- 用户上传内容、附件地址和对象存储凭据。
- 日志、备注和自由文本中可能包含的个人信息。
- 脱敏后主外键、唯一约束和必要业务关系仍然有效。

脱敏结果应由数据负责人确认。本文不提供通用“替换几列即可”的脱敏脚本，因为不同数据模型的敏感字段和关联关系不同。

### 11.3 维护者：导出脱敏副本

在 Windows PowerShell 中确认 `mysqldump.exe` 来自与 `{{MYSQL_MAJOR_VERSION}}` 兼容的 MySQL 客户端：

~~~powershell
mysqldump.exe --version
mysql.exe --host="{{TEST_DB_HOST}}" --port={{TEST_DB_PORT}} --user="{{TEST_DB_EXPORT_USER}}" --password --execute="SELECT VERSION();"
~~~

在 Windows PowerShell 中执行逻辑导出。`--result-file` 让 `mysqldump` 直接写文件，可避免 Windows PowerShell 版本差异导致的重定向编码问题：

~~~powershell
$SnapshotDir = Join-Path $env:USERPROFILE "Downloads\{{PROJECT_DIR}}-db-snapshot"
$SqlFile = "{{PROJECT_DIR}}-local-db-YYYYMMDD-vN.sql"

New-Item -ItemType Directory -Force -Path $SnapshotDir | Out-Null

$DumpArgs = @(
  "--host={{TEST_DB_HOST}}"
  "--port={{TEST_DB_PORT}}"
  "--user={{TEST_DB_EXPORT_USER}}"
  "--password"
  "--single-transaction"
  "--quick"
  "--routines"
  "--triggers"
  "--events"
  "--hex-blob"
  "--default-character-set=utf8mb4"
  "--set-gtid-purged=OFF"
  "--no-tablespaces"
  "--databases"
  "{{MYSQL_DATABASE}}"
  "--result-file=$SnapshotDir\$SqlFile"
)

& mysqldump.exe @DumpArgs

if ($LASTEXITCODE -ne 0) {
  throw "mysqldump failed with exit code $LASTEXITCODE"
}

Get-Item (Join-Path $SnapshotDir $SqlFile)
~~~

说明：

- `--single-transaction` 为 InnoDB 表提供一致性读取；导出期间仍应避免 DDL 变更。
- `--routines`、`--triggers` 和 `--events` 保留数据库对象。
- `--set-gtid-purged=OFF` 避免把测试环境的 GTID 状态带入个人本地库。
- `--no-tablespaces` 可避免本地开发快照依赖额外的表空间权限。
- 如果测试库不是兼容的 MySQL 版本，应由数据库负责人调整参数，不能盲目删掉失败选项后继续分发。

### 11.4 维护者：压缩、校验并发布

在维护者自己的 WSL 中，将 SQL 文件复制到 Linux 用户目录后压缩：

~~~bash
snapshot_dir="$HOME/.local/share/{{PROJECT_DIR}}/db-snapshots"
sql_file="{{PROJECT_DIR}}-local-db-YYYYMMDD-vN.sql"
windows_sql="/mnt/c/Users/{{WINDOWS_USER_NAME}}/Downloads/{{PROJECT_DIR}}-db-snapshot/$sql_file"

mkdir -p "$snapshot_dir"
chmod 700 "$snapshot_dir"
install -m 600 "$windows_sql" "$snapshot_dir/$sql_file"

gzip -9 "$snapshot_dir/$sql_file"

cd "$snapshot_dir"
sha256sum "$sql_file.gz" > "$sql_file.gz.sha256"
gzip -t "$sql_file.gz"
cat "$sql_file.gz.sha256"
~~~

将下列内容上传到 `{{SNAPSHOT_URL}}` 所代表的公司内部平台：

- `.sql.gz` 快照。
- 对应的 `.sha256` 文件。
- 快照说明：导出日期、MySQL 版本、适用的代码提交或迁移版本、数据范围、脱敏审批人、有效期。

不得上传原始未脱敏 SQL，不得把快照提交到 Git，也不得使用个人网盘或公共对象存储。

### 11.5 新员工：下载并校验快照

从公司内部地址 `{{SNAPSHOT_URL}}` 下载：

- `{{SNAPSHOT_FILE_NAME}}`。
- `{{SNAPSHOT_FILE_NAME}}.sha256`。

假设两个文件位于 Windows 下载目录，在 WSL 中执行：

~~~bash
snapshot_dir="$HOME/.local/share/{{PROJECT_DIR}}/db-snapshots"
windows_downloads="/mnt/c/Users/{{WINDOWS_USER_NAME}}/Downloads"

mkdir -p "$snapshot_dir"
chmod 700 "$snapshot_dir"

install -m 600 \
  "$windows_downloads/{{SNAPSHOT_FILE_NAME}}" \
  "$snapshot_dir/{{SNAPSHOT_FILE_NAME}}"

install -m 600 \
  "$windows_downloads/{{SNAPSHOT_FILE_NAME}}.sha256" \
  "$snapshot_dir/{{SNAPSHOT_FILE_NAME}}.sha256"

cd "$snapshot_dir"
sha256sum -c "{{SNAPSHOT_FILE_NAME}}.sha256"
gzip -t "{{SNAPSHOT_FILE_NAME}}"
~~~

`sha256sum` 必须显示 `OK`。同时核对发布说明中的 SHA-256 是否等于：

~~~text
{{SNAPSHOT_SHA256}}
~~~

校验失败时：

1. 不要解压或导入。
2. 删除本次下载文件。
3. 从内部平台重新下载。
4. 仍失败时联系快照维护者。

### 11.6 新员工：导入本地 MySQL

先返回仓库根目录并确认 MySQL 已就绪：

~~~bash
cd "$HOME/workspace/{{PROJECT_DIR}}"
docker compose --env-file .env.local ps
~~~

> **导入会覆盖同名数据库对象**
>
> 该流程用于首次搭建或已明确允许重建的个人本地数据库。导入前如果已有需要保留的本地数据，必须先备份。不要把它指向测试库、共享库或生产库。

启用管道失败检测，并把压缩快照流式导入 MySQL 容器：

~~~bash
set -o pipefail

snapshot_path="$HOME/.local/share/{{PROJECT_DIR}}/db-snapshots/{{SNAPSHOT_FILE_NAME}}"
test -s "$snapshot_path"
gzip -t "$snapshot_path"

gzip -dc "$snapshot_path" |
  docker compose --env-file .env.local exec -T "{{MYSQL_SERVICE}}" \
    sh -eu -c '
      MYSQL_PWD="$(printenv "{{MYSQL_PASSWORD_ENV}}")"
      export MYSQL_PWD
      exec mysql --user=root
    '
~~~

命令无报错返回后，才表示 SQL 流已被 MySQL 客户端完整处理。密码仅从本地容器已有配置中读取；如果项目使用 Docker Secret，应由文档维护者改为仓库现有的 Secret 读取方式。

### 11.7 验证导入结果

执行基础检查和项目约定的关键数据查询：

~~~bash
docker compose --env-file .env.local exec -T "{{MYSQL_SERVICE}}" \
  sh -eu -c '
    MYSQL_PWD="$(printenv "{{MYSQL_PASSWORD_ENV}}")"
    export MYSQL_PWD
    exec mysql --user=root --batch --table --execute="
      SELECT VERSION() AS mysql_version;
      SELECT COUNT(*) AS table_count
      FROM information_schema.tables
      WHERE table_schema = '\''{{MYSQL_DATABASE}}'\'';
      {{KEY_DATA_VALIDATION_SQL}}
    "
  '
~~~

确认：

- MySQL 主版本与 `{{MYSQL_MAJOR_VERSION}}` 兼容。
- `{{MYSQL_DATABASE}}` 中存在预期业务表。
- `{{KEY_DATA_VALIDATION_SQL}}` 返回维护者给出的预期结果。
- 启动后端后，应用可以读取这些数据完成一条关键业务查询。

### 11.8 什么时候可以删除本地卷

只有同时满足以下条件，才可以重建 Compose 项目的全部本地数据：

- 当前目录确认是个人本地项目。
- MySQL、Redis 和其他 Compose 卷都允许丢弃。
- 已备份需要保留的个人调试数据。
- 没有其他项目复用这些卷。

确认无误后才可执行：

~~~bash
docker compose --env-file .env.local down -v
docker compose --env-file .env.local up -d \
  "{{MYSQL_SERVICE}}" \
  "{{REDIS_SERVICE}}"
~~~

> **危险：`down -v` 不可恢复地删除当前 Compose 项目的命名卷。**
>
> 它不是解决连接失败、密码错误或导入失败的默认手段。执行后需要重新等待 MySQL 初始化并再次导入快照。

### 11.9 为什么首版不自动挂载快照

MySQL 官方镜像只会在空数据目录首次初始化时执行 `/docker-entrypoint-initdb.d` 中的 `.sql`、`.sql.gz` 等文件。已有数据卷不会重复执行这些文件。

本流程先采用“下载 → 校验 → 手工导入”，原因是：

- 不需要修改现有 `compose.yaml`。
- 导入前可以明确校验快照来源和哈希。
- 新员工能看到导入失败位置。
- 不会误以为每次 `docker compose up` 都会刷新数据库。

如果团队后续希望自动化，应单独设计快照版本、空卷判断、脱敏数据获取、凭据和失败重试流程。

## 十四、标准开发与 MR 流程

### 14.1 开始一个新任务

先确认当前工作区没有未处理改动：

~~~bash
cd "$HOME/workspace/{{PROJECT_DIR}}"
git status
~~~

如果工作区不干净，先确认改动属于哪个任务。不要用 `git reset --hard` 或 `git clean -fd` 直接清理来源不明的内容。

更新 `develop`：

~~~bash
git fetch origin --prune
git switch develop
git pull --ff-only origin develop
git status --short --branch
~~~

从最新 `develop` 创建功能分支：

~~~bash
git switch -c "{{FEATURE_BRANCH_PATTERN}}"
git branch --show-current
~~~

不要直接在 `develop` 上开发或提交。

### 14.2 开发与验证

开发过程中保持后端、前端和依赖服务边界清楚：

- 后端代码在 `java_service`。
- 前端代码在 `vue_service`。
- MySQL、Redis 由仓库根目录的 `compose.yaml` 管理。
- `.env.local`、数据库快照、Maven 凭据和 IDE 个人配置不得进入提交。

提交前至少执行：

~~~bash
git status
git diff

cd "$HOME/workspace/{{PROJECT_DIR}}/java_service"
{{BACKEND_BASELINE_COMMAND}}

cd "$HOME/workspace/{{PROJECT_DIR}}/vue_service"
# 执行项目 package.json 中规定的检查、测试或构建命令

cd "$HOME/workspace/{{PROJECT_DIR}}"
git diff --check
~~~

### 14.3 提交、推送并创建 MR

只暂存本任务相关文件：

~~~bash
git status
git add path/to/changed-file
git diff --staged
git diff --check --staged
git commit -m "<类型>: <简洁说明>"
~~~

首次推送：

~~~bash
git push -u origin "$(git branch --show-current)"
~~~

在公司代码平台创建 MR：

- Source branch：当前功能分支。
- Target branch：`develop`。
- 标题和描述说明改动目标、主要实现和风险。
- 写明已执行的验证命令及结果。
- 标明未执行的检查及原因。
- 涉及数据库、配置或接口变更时，补充迁移和回滚说明。

根据评审意见修改后，重新执行对应验证并继续推送同一功能分支。不要为了“让 MR 变绿”跳过失败测试或关闭安全校验。

## 十五、每天开始和结束开发

### 15.1 每天开始

进入 WSL 和项目：

~~~powershell
wsl -d "{{WSL_DISTRO_NAME}}"
~~~

~~~bash
cd "$HOME/workspace/{{PROJECT_DIR}}"

git status
git branch --show-current
git fetch origin --prune

docker compose --env-file .env.local up -d \
  "{{MYSQL_SERVICE}}" \
  "{{REDIS_SERVICE}}"

docker compose --env-file .env.local ps
~~~

如果是继续已有任务，留在当前功能分支；如果开始新任务，按“开始一个新任务”从最新 `develop` 创建新分支。

随后分别打开：

~~~bash
cd "$HOME/workspace/{{PROJECT_DIR}}/java_service"
# 使用 IDEA 打开并启动
~~~

~~~bash
cd "$HOME/workspace/{{PROJECT_DIR}}/vue_service"
nvm use 24.13.0
code .
~~~

### 15.2 每天结束

先保存和检查代码：

~~~bash
cd "$HOME/workspace/{{PROJECT_DIR}}"
git status
git diff
~~~

停止本地依赖但保留数据：

~~~bash
docker compose --env-file .env.local stop
~~~

不要在日常结束时执行 `docker compose down -v`。

如果要关闭全部 WSL 实例，在 Windows PowerShell 中执行：

~~~powershell
wsl --shutdown
~~~

关闭 WSL 会停止其中的 Docker daemon 和容器。下次进入 WSL 后，按“每天开始”重新启动服务。

## 十六、常见问题

### WSL 中没有代理变量

1. 确认 Windows 版本满足 autoProxy 要求。
2. 确认 `%UserProfile%\.wslconfig` 中的 `[wsl2]` 和 `autoProxy=true` 拼写正确。
3. 执行 `wsl --shutdown` 后重新进入。
4. 确认 Windows 代理本身已启用。
5. 若公司使用 PAC、认证代理或企业证书，联系 IT 获取专用步骤。

### `sudo apt update` 超时

先运行 `sudo true`，区分 sudo 本身卡住还是提权后的进程缺少代理。如果普通用户的 `curl` 正常，且 `sudo env` 中没有代理变量，按“让 sudo 安全继承代理变量”创建 `/etc/sudoers.d/wsl-proxy-env`。不要把网络超时误判为 Ubuntu 软件包不存在，也不要用 `sudo -E` 扩大环境变量继承范围。

### Maven 无法下载依赖

执行：

~~~bash
mvn -version
stat -c '%A %n' "$HOME/.m2/settings.xml"
~~~

确认 Maven 3.9.0、Java 17 和 `settings.xml` 路径正确。`401`、`403`、`407`、TLS 和 PKIX 错误分别对应仓库权限、代理认证或证书问题，不要删除整个 `~/.m2/repository` 来掩盖根因。

### nvm 存在但 `node` 版本错误

~~~bash
export NVM_DIR="$HOME/.nvm"
[ -s "$NVM_DIR/nvm.sh" ] && . "$NVM_DIR/nvm.sh"
nvm use 24.13.0
nvm alias default 24.13.0
node --version
~~~

### `docker pull` 失败但 `curl` 正常

检查 Docker daemon 代理、企业根证书、DNS 和镜像仓权限。Shell 代理成功不能证明 daemon 已配置。查看：

~~~bash
sudo systemctl show --property=Environment docker
sudo journalctl -u docker.service -n 100 --no-pager
docker info
~~~

不要写入来源不明的公共镜像加速地址。

### MySQL 长时间没有就绪

~~~bash
docker compose --env-file .env.local ps
docker compose --env-file .env.local logs \
  --tail 200 \
  "{{MYSQL_SERVICE}}"
~~~

重点检查端口冲突、磁盘空间、密码变量、数据卷权限、初始化日志和 MySQL 镜像版本。

### 快照导入提示 `Access denied`

确认：

- `{{MYSQL_PASSWORD_ENV}}` 是 MySQL 容器内真实存在的本地密码变量。
- 导入目标是个人本地容器。
- `.env.local` 与当前 Compose 项目匹配。
- 没有把测试库账号误用于本地容器。

不要把密码直接追加到命令行参数或粘贴到聊天中。

### 快照导入出现 GTID、字符集或排序规则错误

通常是导出客户端、源数据库和本地容器版本不兼容。记录：

~~~bash
docker compose --env-file .env.local exec -T "{{MYSQL_SERVICE}}" \
  mysql --version
~~~

将完整错误、源版本、目标版本和快照版本交给维护者重新生成兼容快照，不要在个人电脑上批量删除不理解的 SQL。

### 快照导入出现 `DEFINER` 或权限错误

视图、触发器、函数或事件可能引用测试环境账户。应由快照维护者在受控流程中移除或映射不适用于本地环境的 DEFINER，并重新发布快照。不要为了导入而给个人本地应用账号授予不必要的 `SUPER` 等高权限。

### 修改初始化文件后数据库没有变化

MySQL 官方镜像只会在空数据目录首次初始化时运行 `/docker-entrypoint-initdb.d`。已有卷不会因为容器重启而重新执行初始化文件。先判断是否确实允许丢弃本地数据，再决定是否按危险步骤重建卷。

### IDEA 能打开项目但构建工具不一致

在 IDEA Terminal 中执行：

~~~bash
which java
which mvn
java -version
mvn -version
~~~

它们都应来自 WSL。若 IDEA 使用 Windows JDK、内置 Maven 或错误的 `settings.xml`，重新配置项目 SDK、Maven home 和 importer JDK。

### VS Code 使用了 Windows Node.js

确认左下角显示 `WSL: {{WSL_DISTRO_NAME}}`，集成终端中的 `pwd` 位于 `/home/...`。从 WSL 终端执行 `code .`，不要从 Windows 文件资源管理器用普通本地窗口打开 `vue_service`。
