本文说明 UTM 中 Linux 开发虚拟机的日常启动、暂停、重启和关机方式。重点是区分 UTM 窗口、虚拟机运行状态和客户机操作系统状态，优先让 Ubuntu 正常完成服务停止、文件系统卸载和关机，只在系统真正失去响应时使用强制操作。

> [!info] 资料核对日期
> 本文涉及 UTM 控制按钮、关机方式和暂停限制的信息于 **2026-07-20** 根据官方文档核对。UTM 的按钮图标、文字和可用能力会随版本、虚拟化后端及虚拟设备变化，实际操作时应重新核对文末官方资料。

## 完成标准

完成本文后，应能够：

- 区分关闭 UTM 窗口、停止虚拟机进程和让 Ubuntu 正常关机。
- 正常启动虚拟机，并判断客户机是否已经可以使用。
- 从 Ubuntu 内正常重启或关机。
- 解释 Request Power Down、Force Shut Down 和 Force Kill 的风险差异。
- 判断什么时候可以暂停，什么时候必须完整关机。
- 在提示符消失或客户机疑似卡住时按低风险顺序排查。

## 1. 先区分三个层次

一台正在 UTM 中运行的 Linux 虚拟机至少涉及三个层次：

| 层次 | 作用 | 关闭或停止的含义 |
| --- | --- | --- |
| UTM 窗口 | 在 macOS 上显示和控制虚拟机 | 窗口消失不一定表示客户机已经正常关机 |
| 虚拟机进程 | 提供虚拟 CPU、内存、磁盘和网卡 | 强制终止类似直接切断实体机电源 |
| Ubuntu 客户机 | 运行 Shell、服务、容器和文件系统 | 正常关机会依次停止服务、写回数据并卸载文件系统 |

因此，“关闭窗口”“停止 VM”和“Ubuntu 已正常关机”不能视为同一件事。日常使用应先在客户机内发起关机，再等待 UTM 明确显示虚拟机已经停止。

UTM 会区分以下运行状态，界面文字可能随版本变化：

| 状态 | 含义 | 是否仍占用运行资源 |
| --- | --- | --- |
| Stopped | 虚拟机进程已经停止 | 不再占用客户机运行内存和 CPU |
| Starting / Started | 正在启动或运行 | 是 |
| Pausing / Paused | 正在暂停或已经暂停 | 客户机不继续执行，但这不等于正常关机 |
| Resuming | 正在从暂停状态恢复 | 是 |
| Stopping | 正在停止 | 等待状态变为 Stopped，不要反复点击控制按钮 |

## 2. 正常启动虚拟机

1. 打开 UTM，在虚拟机列表中选择目标 VM。
2. 点击启动按钮一次。
3. 等待 Ubuntu 完成启动，不要因为控制台暂时没有提示符而连续重置或强制关闭。
4. 出现登录提示后使用普通用户登录；如果平时通过 SSH 使用，也可以从宿主机建立新的 SSH 会话。

登录后可做最小只读检查。

**执行位置：Ubuntu 虚拟机（任意目录）**

```bash
systemctl is-system-running
systemctl --failed --no-pager
uptime -s
```

常见结果：

- `running`：systemd 认为系统已经完成启动，且没有失败的 unit。
- `degraded`：系统通常仍能使用，但至少有一个 unit 失败，应按 [[systemd 服务与日志基础]] 检查失败项。
- `starting`：系统仍在启动，等待后重新检查。
- `systemctl --failed` 没有列出失败项：当前没有处于 failed 状态的 unit。

一次启动中出现过警告，不等于虚拟机已经损坏。先判断系统、网络和实际需要的服务是否可用，再针对失败项读取日志。

## 3. 正常重启与关机

重启或关机前：

1. 保存编辑器、Shell 和远程开发会话中的工作。
2. 等待正在写入磁盘的构建、导入或升级完成。
3. 对需要显式保持一致性的应用或数据库，使用它们自己的停止、备份或维护流程。

正常重启：

**执行位置：Ubuntu 虚拟机（任意目录）**

```bash
sudo systemctl reboot
```

正常关机：

**执行位置：Ubuntu 虚拟机（任意目录）**

```bash
sudo systemctl poweroff
```

也可以使用 Ubuntu 中常见的等价关机命令：

```bash
sudo shutdown -h now
```

通过 SSH 执行重启或关机后，当前连接会断开，这是预期结果。关机后回到 UTM，等待状态明确变为 **Stopped**，再退出 UTM、修改虚拟硬件或复制虚拟机。

> [!warning] 不要为追求“更快”添加强制参数
> `reboot -f`、`poweroff -f` 或多重 `systemctl --force` 可能绕过正常的 init 和文件系统处理，不属于日常启停命令。

## 4. UTM 电源操作的区别

UTM 官方建议优先从客户机操作系统内关机。在 macOS 上长按电源按钮可以看到额外的关机选项；直接点击电源按钮的默认操作可能是强制关闭，因此不能把它当作日常关机入口。

| UTM 操作 | 实际含义 | 使用建议 |
| --- | --- | --- |
| Request Power Down | 向客户机发送电源关闭请求，类似按一下实体机电源键；客户机可能忽略 | 无法进入 Shell，但客户机仍可能响应时使用 |
| Force Shut Down | 要求虚拟机进程强制停止，类似长按实体机电源键 | 正常关机和请求关机均无效时才使用 |
| Force Kill | 直接终止虚拟机后端进程 | 风险最高，只作为最后手段 |
| Reset | 重置虚拟机，类似实体机硬重启 | 会丢弃未保存状态，不替代正常重启 |

无法从 Shell 正常关机时，按以下顺序处理：

1. 等待当前磁盘操作或系统更新完成。
2. 尝试通过 UTM 控制台或新的 SSH 会话登录，并执行 `sudo systemctl poweroff`。
3. 长按 UTM 电源按钮，选择 **Request Power Down**，然后等待客户机处理。
4. 只有客户机持续无响应时，才使用 **Force Shut Down**。
5. 只有虚拟机后端也无法停止时，才使用 **Force Kill**。

不要连续点击多个关机动作。进入 **Stopping** 状态后先等待，避免在正常清理过程中再次强制终止。

## 5. 暂停不等于关机

暂停会停止客户机继续执行，但不会让 Ubuntu 完成正常关机流程，也不能替代备份前的一致性处理。

暂停适合：

- 短时间离开，稍后希望从原来的会话继续。
- 当前 UTM 后端和虚拟设备明确支持暂停与恢复。
- 没有把暂停状态当作独立备份或长期恢复方案。

完整关机适合：

- 需要退出 UTM、重启宿主机或长时间不用虚拟机。
- 需要修改 vCPU、内存、磁盘或其他虚拟硬件。
- 需要创建关机状态的完整 VM 副本。
- 正在排查启动、文件系统或设备状态问题。

UTM 官方说明，暂停可能因虚拟设备或后端限制而不可用，例如某些模拟 NVMe 设备、GPU 加速显示设备或特定 Intel QEMU 场景。按钮不可用时不要假设系统故障，也不要用强制关机模拟暂停。

## 6. 关闭窗口时如何判断是否安全

关闭 UTM 窗口是宿主机界面操作，不是 Ubuntu 关机命令。若关闭窗口或退出 UTM 时出现确认提示，应读取它要执行的具体动作，不要只根据“关闭”“停止”一类概括文字判断是否为正常关机。

最可靠的日常流程是：

1. 在 Ubuntu 内执行 `sudo systemctl poweroff`。
2. 等待控制台结束。
3. 回到 UTM 列表，确认 VM 状态为 **Stopped**。
4. 再关闭窗口或退出 UTM。

不要把“UTM 窗口已经消失”当作客户机正常关机的证据。

## 7. 提示符消失不一定是卡死

控制台只有光标闪烁、看不到 `user@host:~$` 时，先按一次 Enter 观察，不要立即重置虚拟机。

常见情况：

- 启动日志在提示符之后继续输出：按一次 Enter 通常会重新显示提示符。
- 画面出现 `UNIT LOAD ACTIVE SUB DESCRIPTION`，底部有 `(lines ...)`：当前处于 `systemctl` 输出的分页器中，按 `q` 退出；Enter 只会继续滚动。
- 某个前台命令仍在运行：先阅读当前内容；明确可以中断时使用 `Ctrl+C`。
- 控制台没有响应但 SSH 仍可登录：优先从 SSH 检查系统并正常关机。
- 控制台和 SSH 都持续无响应：再按上一节的请求关机、强制关机、强制终止顺序处理。

判断“虚拟机卡死”需要结合键盘响应、SSH、系统日志和服务状态，不能只根据提示符暂时不可见得出结论。

## 8. 强制关闭后如何检查

强制关闭不代表文件系统必然损坏，但下一次启动后应先确认系统和关键服务状态。

**执行位置：Ubuntu 虚拟机（任意目录）**

```bash
systemctl is-system-running
systemctl --failed --no-pager
journalctl -b -p warning..alert --no-pager -n 100
```

最后一条会显示本次启动中最近 100 条 warning 到 alert 级别的日志，出现输出不等于每一项都需要修复。优先处理文件系统、I/O、网络、容器、数据库或项目必需服务的明确失败，不要因为普通告警直接重装系统或对已挂载的根文件系统运行 `fsck`。

若强制关闭发生在系统升级、数据库写入或备份过程中，应按对应专题验证数据和服务。创建独立 VM 副本、源码备份和数据库备份的流程见 [[UTM 虚拟机快照、备份与恢复]]。

## 日常检查清单

- [ ] 启动按钮只点击一次，并等待 Ubuntu 完成启动。
- [ ] 日常重启使用 `sudo systemctl reboot`。
- [ ] 日常关机使用 `sudo systemctl poweroff`，并等待 UTM 显示 Stopped。
- [ ] 能区分暂停、请求关机、强制关机和强制终止。
- [ ] 提示符消失时先判断是否为日志输出、分页器或前台命令。
- [ ] 只有客户机真正无响应时才逐级使用强制操作。
