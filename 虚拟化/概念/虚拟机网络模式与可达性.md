本文从“谁需要访问谁”出发，解释虚拟机常见的 Shared、NAT、Bridged、Host-only 和 Isolated 等网络模式，以及端口转发、覆盖网络、服务监听和防火墙与这些模式的关系。目标不是记住某款虚拟化软件的下拉菜单，而是能根据访问方向、隔离边界和运行环境作出选择。

不同产品可能对相似机制使用不同名称，同名模式的实现细节也未必完全一致。理解通用模型后，仍应核对目标产品、虚拟化后端和当前版本的官方说明。

开始前先理解 [[虚拟机、客户机与 CPU 架构]]。Linux 客户机如何管理接口、IP 地址、路由和 DNS，见 [[Linux 网络接口、IP 地址、路由与 DNS 基础]]。

## 1. 先区分不同网络层

一次从客户端到虚拟机服务的连接至少涉及以下层次：

| 层次 | 常见对象 | 负责什么 |
| --- | --- | --- |
| 宿主机所在网络 | Wi-Fi、以太网、路由器、公司或家庭网络 | 决定宿主机自身能到达哪些网络 |
| 虚拟化网络 | Shared、NAT、Bridged、Host-only、Isolated | 决定虚拟网卡被接到哪里，以及默认允许哪些方向 |
| 客户机网络 | 虚拟网卡、DHCP、静态地址、路由、DNS | 让客户机在所接入的网络中取得并使用地址 |
| 应用与访问控制 | `sshd`、监听地址、端口、客户机防火墙 | 决定某项服务是否真正接受连接 |
| 可选覆盖网络 | Tailscale 等软件创建的接口和地址 | 在已有基础网络之上增加另一条受控访问路径 |

虚拟化软件负责把虚拟网卡接入某种网络；客户机操作系统仍负责接口、IP 地址、路由和 DNS；应用还必须监听正确的地址和端口。任何一层正常，都不能单独证明整条访问路径正常。

## 2. 先明确访问方向

“虚拟机已经联网”通常只回答了客户机能否主动向外访问，并不等于其他设备能够反向访问客户机。

| 访问方向 | 典型需求 | 需要回答的问题 |
| --- | --- | --- |
| 客户机 → 互联网 | 更新系统、下载依赖、拉取镜像 | 客户机是否有地址、默认路由、DNS 和出站路径？ |
| 宿主机 → 客户机 | 本机 SSH、访问开发服务 | 宿主与客户机是否直接互通，服务是否监听并放行？ |
| 局域网设备 → 客户机 | 其他电脑或手机访问 VM 服务 | 客户机是否在 LAN 可达，或是否另有受控访问路径？ |
| 客户机 ↔ 客户机 | 多节点实验、数据库与应用分离 | 两台 VM 是否接入同一个允许互通的虚拟网络？ |
| 其他网络 → 客户机 | 外出访问、跨办公地点访问 | 使用覆盖网络、VPN，还是经过严格设计的公网入口？ |

先写清方向，再讨论模式。不要根据“客户机能访问互联网”反推出“局域网设备也能访问客户机”。

## 3. 常见网络接入模式

### Shared 或宿主路由网络

这类模式由宿主操作系统或虚拟化框架为客户机路由流量。客户机通常能够出网，宿主机与客户机往往也能互相访问；局域网其他设备是否能够直接访问客户机，则取决于产品实现和宿主网络策略。

它适合作为开发虚拟机的起点，因为通常不需要把客户机直接暴露为局域网中的独立设备。Shared 与传统 NAT 的实现并不必然相同，不应只因为使用体验相似就把两个名称当成严格同义词。

### NAT 或用户态网络

客户机通常位于虚拟私有子网中，主动发出的流量由宿主转换或代理后再访问外部网络。外部设备默认不能任意发起到客户机的连接；若产品支持，可以为特定 TCP 或 UDP 端口建立转发。

这类模式适合宿主网络经常变化的笔记本电脑、只需要客户机出网的环境，以及希望显式控制入站端口的场景。不同产品可能使用 NAT、user-mode networking、Emulated VLAN 等不同名称。

### Bridged 桥接网络

桥接把客户机的虚拟网卡接到宿主机选定的二层网络。客户机通常像局域网中的独立设备一样取得地址，并能被同一局域网中的其他设备直接寻址。

桥接适合确实需要 LAN 其他设备直接访问客户机的场景，但会扩大客户机的网络暴露面。Wi-Fi、企业网络准入、交换机策略、MAC 地址限制和宿主平台能力都可能限制桥接；不能因为界面允许选择，就假设所在网络一定接受。

### Host-only 仅宿主网络

Host-only 通常建立只供宿主机和一组客户机使用的私有网络，默认不提供到外部网络的路径。它适合需要宿主访问客户机、客户机之间互通，但不希望客户机直接出网的实验环境。

是否自带 DHCP、不同虚拟机是否默认互通以及是否允许额外路由，取决于具体产品。

### Isolated 或 Internal 隔离网络

隔离网络通常只允许接入同一虚拟网络的客户机互通，不提供外部网络路径；有些实现也会把宿主机排除在外。它适合多节点协议实验、恶意样本分析前的隔离环境或不应接触外网的测试系统。

“Host-only”和“Isolated”在不同产品中的边界可能不同，必须以实际产品说明和连通性验证为准。

## 4. 典型可达性对比

下表描述常见默认行为，不代替具体产品文档。这里的“通常”仍可能被宿主网络、产品设置、防火墙和服务监听改变。

| 模式 | 客户机出网 | 宿主机访问客户机 | LAN 其他设备访问客户机 | 同一虚拟网络内 VM 互通 | 常见地址来源 |
| --- | --- | --- | --- | --- | --- |
| Shared / 宿主路由 | 通常可以 | 通常可以 | 通常不直接提供 | 取决于产品 | 虚拟化框架或宿主网络 |
| NAT / 用户态网络 | 通常可以 | 可能需要产品提供的直达路径或端口转发 | 默认不可以 | 取决于产品 | 虚拟 DHCP |
| Bridged | 取决于所在 LAN | 通常可以 | 通常可以 | 与普通 LAN 设备相同 | LAN DHCP 或手动配置 |
| Host-only | 默认不可以 | 通常可以 | 默认不可以 | 通常可以 | 私有 DHCP 或手动配置 |
| Isolated / Internal | 不可以 | 取决于产品，常见为不可以 | 不可以 | 通常可以 | 虚拟 DHCP 或手动配置 |

即使表格显示某个方向“通常可以”，也仍需满足以下条件：目标地址可路由、应用正在运行、应用未只监听回环地址、客户机防火墙放行，以及额外访问控制允许连接。

## 5. 端口转发和覆盖网络不是接入模式

### 端口转发

端口转发是在既有网络模式上增加一条特定入口，例如把宿主机的某个 TCP 端口转发到客户机的 SSH 端口。它只改变明确配置的协议与端口，不会让客户机整体变成局域网中的独立设备。

配置前需要确认：

- 当前虚拟化后端和网络模式是否支持端口转发。
- 宿主监听地址是仅回环地址、某块网卡地址，还是所有地址。
- 宿主端口是否冲突。
- 客户机地址和目标端口是否正确。
- 客户机服务是否监听，防火墙是否放行。

把宿主监听地址从回环地址扩大到所有接口，会扩大可访问范围。不要为了“先连通”而忽略暴露面变化。

### 覆盖网络

Tailscale 一类软件会在宿主机或客户机操作系统中创建额外接口、地址和受身份控制的路径。它依赖底层网络先能传输数据，但不要求把虚拟机改成桥接模式，也不会替代 UTM、DHCP、客户机路由或应用服务。

覆盖网络可能在节点间建立直接连接，也可能在直接连接不可用时经过中继。它解决的是跨局域网或复杂 NAT 环境下的受控可达性，不是虚拟网卡接入宿主网络的方式。具体使用见 [[使用 Tailscale 访问 Linux 主机]]。

### 服务监听与防火墙

网络模式只提供路径，不会自动启动服务或开放端口。例如，服务只监听 `127.0.0.1` 时，即使客户机使用桥接网络，其他设备也无法通过客户机网卡地址访问它。

因此，“选择桥接”“配置端口转发”“开放防火墙”和“让服务监听外部接口”是不同操作，应分别说明目的并逐层验证。

## 6. UTM 中的名称映射

UTM 可以使用 Apple 虚拟化后端或 QEMU 后端，两者展示的网络模式并不完全相同。先确认当前虚拟机使用的后端，再阅读对应设置说明。

| UTM 后端 | UTM 模式 | 对应的典型行为 |
| --- | --- | --- |
| Apple | Shared Network | 宿主系统直接路由流量；宿主与客户机服务可互相访问；UTM 推荐新虚拟机使用 |
| Apple | Bridged | 通过指定接口建立二层桥接；Wi-Fi 可能需要额外配置 |
| QEMU | Emulated VLAN | 在用户态为该 VM 创建 VLAN；请求在宿主侧表现为来自 UTM 进程 |
| QEMU | macOS Shared Network | 由 macOS 路由，客户机与宿主共享虚拟网络 |
| QEMU | macOS Host Only | 与 Shared 类似但阻止 WAN 流量；自定义 Host Network 的地址与 DHCP 规则需单独核对 |
| QEMU | macOS Bridged | 通过指定宿主接口接入二层网络 |

> [!important] UTM 端口转发的后端限制
> UTM 官方文档说明，内置端口转发只适用于使用 **Emulated VLAN** 的 **QEMU 后端**虚拟机，支持 TCP 和 UDP。不要把它描述成 Apple 后端 Shared Network 的通用附属功能。

对普通本地开发虚拟机，可以先使用 Shared Network 完成客户机出网和宿主机访问客户机；只有明确需要 LAN 其他设备直接访问时才评估桥接。跨局域网访问可在基础路径稳定后单独增加 Tailscale，而不是同时改变桥接、端口转发和防火墙。

## 7. 按需求选择

1. 写出必要的访问方向，不从模式名称开始。
2. 选择能够满足这些方向且暴露范围最小的模式。
3. 核对虚拟化产品、后端和宿主网络是否支持该模式。
4. 明确地址由谁分配：虚拟 DHCP、LAN DHCP，还是经过规划的手动配置。
5. 先验证基础模式，再单独增加端口转发或覆盖网络。
6. 每次只改变一层，并保留控制台或其他回退路径。

常见判断示例：

- 只需要 macOS 宿主机 SSH 到 Ubuntu VM：先使用 UTM Shared Network，不必仅为 SSH 改成桥接。
- 需要同一 LAN 的其他设备直接访问 VM：确认网络策略后评估 Bridged，或为相关设备建立受控覆盖网络。
- 只需要暴露 QEMU Emulated VLAN 中的一项服务：确认后端支持后评估单端口转发。
- 需要外出访问开发 VM：先稳定本地网络和 OpenSSH，再按 [[使用 Tailscale 访问 Linux 主机]] 增加独立路径。
- 需要不接触外网的多节点实验：评估 Host-only 或 Isolated，并实测宿主与 VM 之间的边界。

## 8. 分层验证与排查

先在客户机读取状态，不要立即修改配置。

**执行位置：Linux 客户机（任意目录）**

```bash
ip -brief link
ip -brief address
ip route
getent ahosts archive.ubuntu.com
curl -I --max-time 15 https://archive.ubuntu.com/
ss -lnt
```

再从需要发起访问的宿主机或 LAN 客户端验证目标服务。

**执行位置：访问端（任意目录）**

```bash
printf '客户机地址或可解析名称: '
IFS= read -r VM_HOST
nc -vz "$VM_HOST" 22
```

按以下顺序判断：

1. 虚拟网卡是否存在并接入预期模式。
2. 客户机接口是否启用并取得预期地址。
3. 默认路由、DNS 和出站 HTTPS 是否正常。
4. 当前模式是否本来就允许目标访问方向。
5. 目标服务是否运行并监听客户机网卡地址。
6. 客户机防火墙、宿主防火墙和网络策略是否允许连接。
7. 若使用端口转发或覆盖网络，再单独验证该层的映射、身份和访问控制。

`ping` 失败不能单独证明网络不可用，因为 ICMP 可能被禁用，用户态网络对 ICMP 的支持也可能有限。应使用目标协议验证目标服务，例如用 `nc` 检查 SSH 的 TCP 端口。Linux 地址、路由、DNS 与 Netplan 的详细读取方法见 [[Linux 网络接口、IP 地址、路由与 DNS 基础]]；SSH 服务和身份问题见 [[OpenSSH 连接、密钥与主机指纹]]。

## 9. 常见误区

- **客户机能上网，所以局域网一定能访问它**：出站路径与入站路径不是同一件事。
- **桥接后服务自然可用**：服务仍可能未启动、只监听回环地址或被防火墙阻止。
- **端口转发是一种网络模式**：它是在特定基础模式上增加的单端口入口。
- **Tailscale 替代基础网络**：它仍依赖底层网络，并作为客户机中的另一条访问路径运行。
- **设置静态 IP 就能改变可达性**：静态地址不会自动改变虚拟化模式、路由边界、服务监听或防火墙。
- **同名模式在不同产品中完全等价**：应以目标后端文档和实际连通性为准。
