本文维护 [[现代终端环境搭建概览]] 中的 Ghostty、Zsh、Antidote、Starship、Atuin、zoxide 与 fzf。目标不是追求“一键全部最新”，而是让每次更新只有一个明确变量、一个验证结果和一个可恢复的回退点。

各组件的配置细节分别见 [[Ghostty 常用配置与 Shell 集成]]、[[Zsh 与 Antidote 跨机器配置管理]]、[[Starship 提示符配置]]、[[Atuin 命令历史管理]] 与 [[zoxide 与 fzf 导航和模糊查找]]。从 Oh My Zsh 迁移时，先按 [[从 Oh My Zsh 迁移到 Antidote]] 保留旧环境，验证完成后再清理。

## 1. 维护原则

1. 一次只更新一个层级：终端模拟器、Shell、插件、提示符或单个 CLI 工具。
2. 更新前保留一个正常的终端窗口或 SSH 会话；新窗口通过全部验证后再关闭旧窗口。
3. 二进制始终按原安装方式更新。Homebrew 安装的软件用 Homebrew，APT 软件用 APT，官方安装器安装的软件按该项目说明更新。
4. 配置、插件版本和本地数据是三个不同的回退面，必须分别处理。
5. 不把插件缓存、历史数据库、加密密钥和机器私有配置提交到 Git。

## 2. 更新前建立基线

### 记录当前版本

在正常的交互式 Zsh 中执行：

~~~zsh
zsh --version

if command -v ghostty >/dev/null 2>&1; then
  ghostty +version
fi

if command -v antidote >/dev/null 2>&1; then
  antidote --version
fi

for tool in starship atuin zoxide fzf; do
  if command -v "$tool" >/dev/null 2>&1; then
    "$tool" --version
  else
    printf 'missing: %s\n' "$tool"
  fi
done
~~~

macOS 图形安装的 Ghostty 若未把 CLI 加入 PATH，可在 Ghostty 的“关于”界面确认版本；不要仅为运行版本命令而重复安装应用。

### 固定可回退的配置

如果配置由 dotfiles Git 仓库管理，先确认工作区内容和本次更新无关的改动：

~~~bash
git status --short
git diff --check
~~~

提交、打标签或复制备份属于你的 dotfiles 工作流；至少应确保下列配置能从一个已知良好版本恢复。

| 内容 | 建议进入 Git | 说明 |
| --- | --- | --- |
| `~/.zshenv` | 是 | 只保留 `ZDOTDIR` 和非交互环境所需的最小 PATH |
| `.zprofile`、`.zshrc` | 是 | 登录环境与交互式工具入口 |
| `.zsh_plugins.txt` | 是 | Antidote 插件声明 |
| 明确保存的 Antidote snapshot | 是 | 用提交 SHA 固定插件版本 |
| `starship.toml` | 是 | 提示符配置 |
| Ghostty `config.ghostty` 与公共片段 | 是 | 机器私有片段应单独排除 |
| 审查后的 Atuin `config.toml` | 是 | 只保留通用偏好和过滤规则 |

以下内容不应进入 Git：

| 内容 | 原因 |
| --- | --- |
| `.zsh_plugins.zsh` | Antidote 生成的静态加载文件，可重新生成 |
| `antidote home` 下的仓库 | 插件克隆和缓存，可由声明或 snapshot 恢复 |
| `.zcompdump*` | Zsh 补全缓存，可重新生成 |
| Atuin `history.db`、`key`、`session` | 包含历史数据、解密密钥和登录会话 |
| zoxide 数据库 | 本机目录及使用评分，应在新机器重新积累或私下迁移 |
| Starship 缓存与日志 | 运行时数据，不是配置来源 |
| 代理、令牌、私有主机名和机器专属路径 | 不适合跨机器公开同步 |

Atuin 的本地数据库和密钥虽然不能进入 Git，但仍可能需要私有备份。默认数据目录是 `~/.local/share/atuin`；覆盖了 XDG 路径时以实际配置为准。备份或恢复该目录前，应先关闭其他交互式 Shell，并临时停用 Atuin 初始化，避免数据库在复制过程中继续写入。备份必须放在受保护的位置。

## 3. 按组件更新

不要在开始时执行 `brew upgrade`、系统全量升级或来源不明的“update all”脚本。先选择下表的一行，更新并完成第 5 节验证，再处理下一行。

| 组件 | 更新方式 | 关键边界 |
| --- | --- | --- |
| Ghostty | macOS 使用应用内稳定通道或原包管理器；Linux 使用原发行包维护方式 | Ghostty 只安装在本地图形客户端，远程服务器无需安装 |
| Zsh | 随 macOS 系统更新，或按 Homebrew/APT 的原安装方式更新 | 更新后先验证启动文件与补全，不要同时改插件 |
| Antidote 本体 | 包管理器安装则用包管理器；Git 安装可由 `antidote update` 一并更新 | 更新插件前先保存 snapshot |
| Antidote 插件 | `antidote update` | 会更新所有未 pin 的 bundle；pin 到 SHA 的插件会跳过 |
| Starship | 原包管理器或官方安装器 | 只更新二进制，不在同一步重写 `starship.toml` |
| Atuin | 包管理器安装则用包管理器；官方安装器安装可用 `atuin update` | 先保护本地数据库、key 和 session |
| zoxide | 按原包管理器或官方安装方式 | 数据库与二进制分开，不因升级删除数据库 |
| fzf | 按原包管理器更新；Git 安装按其官方升级步骤 | 更新后检查 `--zsh` 和按键禁用行为 |

Homebrew 环境也应逐个运行，例如先执行 `brew upgrade fzf` 并验证，而不是把 Ghostty、Zsh 和所有 CLI 一起升级。APT 同理，只升级本次选择的包。

## 4. 正确维护 Antidote

### 保存和恢复 snapshot

更新前显式保存当前已克隆 bundle 的时间点：

~~~zsh
antidote snapshot save
antidote snapshot list
~~~

默认 snapshot 位于 `antidote snapshot home` 显示的目录。若要保存一份可提交到 dotfiles 的稳定快照，可指定文件：

~~~zsh
snapshot_dir="${ZDOTDIR:-$HOME}/snapshots"
mkdir -p "$snapshot_dir"
snapshot_file="$snapshot_dir/antidote.txt"
antidote snapshot save "$snapshot_file"
~~~

snapshot 文件把仓库标记为 `kind:clone` 并附带 `pin:<SHA>`，适合跨机器复现；它不包含插件缓存本身。恢复时使用真实文件路径：

~~~zsh
snapshot_file="${ZDOTDIR:-$HOME}/snapshots/antidote.txt"
antidote snapshot restore "$snapshot_file"
~~~

不带文件参数的 `antidote snapshot restore` 在安装了 fzf 时会进入交互选择。自动化或恢复文档应传入明确路径，避免选错快照。

### 更新 bundle

在 snapshot 可用后单独执行：

~~~zsh
antidote update
~~~

该命令会检查 Antidote 自身和所有 bundle；声明中带 `pin:<SHA>` 的 bundle 不会更新。静态模式下，`antidote update` 还会自动保存 snapshot；动态模式不会，因此显式的更新前 snapshot 仍是更清楚的维护习惯。

### 刷新插件清单

当前 Antidote 官方 CLI **没有 `antidote refresh` 子命令**。这里的“刷新”是指修改 `.zsh_plugins.txt` 后重新生成加载文件：

- 使用推荐的 `antidote load` 时，它会在插件清单较新时自动生成静态文件；保存清单后新开终端即可。
- 若采用手工静态模式，使用官方的 `antidote bundle`：

~~~zsh
plugin_base="${ZDOTDIR:-$HOME}/.zsh_plugins"
antidote bundle < "${plugin_base}.txt" >| "${plugin_base}.zsh"
~~~

生成的 `.zsh_plugins.zsh` 是缓存，不应提交 Git。不要同时使用 `antidote load`、手工 `antidote bundle` 和旧式 `antidote init`，否则很难判断实际加载来源。

## 5. 更新后的分层验证

### 第一层：语法检查

先检查固定入口，再让 Zsh 解析 `ZDOTDIR` 并检查实际配置：

~~~bash
if [ -f "$HOME/.zshenv" ]; then
  zsh -n "$HOME/.zshenv"
fi

zsh_config_dir="$(zsh -c 'print -r -- "${ZDOTDIR:-$HOME}"')"

for startup_file in .zprofile .zshrc; do
  if [ -f "$zsh_config_dir/$startup_file" ]; then
    zsh -n "$zsh_config_dir/$startup_file"
  fi
done
~~~

`zsh -n` 只检查可解析性，不会证明被 `source` 的文件存在，也不会证明插件运行正常。

### 第二层：交互、登录与 SSH-like 检查

~~~bash
zsh -ic 'print -r -- "interactive shell: ok"; command -v antidote starship atuin zoxide fzf'
zsh -lic 'print -r -- "login interactive shell: ok"'
env -i HOME="$HOME" USER="$USER" LOGNAME="$LOGNAME" SHELL=/bin/zsh \
  PATH=/usr/bin:/bin:/usr/sbin:/sbin \
  /bin/zsh -c '
    print -r -- "ssh-like PATH=$PATH"
    for tool in git starship atuin zoxide fzf; do
      command -v "$tool" || print -r -- "$tool: not found"
    done
  '
~~~

- `zsh -ic` 验证 `.zshrc` 的交互工具。
- `zsh -lic` 同时验证登录环境和交互配置。
- `env -i ... /bin/zsh -c` 从不继承当前终端的 `PATH` 开始，近似检查远程命令或自动化中的非交互 Shell；输出结果应能说明 `.zshenv` 是否补齐了 Homebrew 等必要二进制路径，但不应要求 Starship、Atuin 或交互插件完成初始化。Antidote 在本方案中由 `.zshrc` 加载为 Shell 函数，因此只在前面的交互式检查中验证，不把它当成非交互 PATH 命令。

若实际维护远程主机，还应执行一次真实的非破坏性命令，例如 `ssh <主机> 'command -v git; printf "%s\n" "$PATH"'`。替换成已知主机后再运行，不要把示例占位符直接复制到脚本。

### 第三层：组件自检

~~~bash
atuin doctor
antidote --version
antidote list
starship explain
~~~

重点检查：

- `atuin doctor` 的 `shell.preexec` 不应为 `none`。
- `Ctrl-R` 只能打开 Atuin，`Ctrl-T` 打开 fzf，`zi` 打开目录选择。
- Starship 在普通目录和大型 Git 仓库中都能及时出现。
- 若 Ghostty CLI 可用，`ghostty +show-config` 能读取当前合并配置；随后在应用内重新加载配置并观察告警。

最后粗略观察启动耗时：

~~~bash
time zsh -i -c exit
~~~

单次耗时会受磁盘缓存影响，适合发现“从瞬时变成明显停顿”，不适合当作严格基准。

## 6. 按层隔离故障

遇到新窗口报错、卡住或无提示符时，按以下顺序缩小范围：

1. **终端层**：用 macOS Terminal、系统控制台或另一款终端启动 Zsh。只有 Ghostty 失败时，先恢复 Ghostty 配置或版本。
2. **Shell 层**：运行 `zsh -f`。无用户配置的 Zsh 正常，说明问题位于启动文件或其加载内容。
3. **配置层**：执行第 5 节的 `zsh -n`，检查真实 `ZDOTDIR`，再看最近一次配置改动。
4. **插件层**：先恢复 Antidote snapshot，或临时移除最后新增的一个 bundle；不要删除整个插件目录。
5. **独立工具层**：分别运行 `atuin doctor`、`starship explain`、`zoxide --version` 和 `fzf --version`，不要把所有异常归因于 Antidote。
6. **按键层**：检查 `.zshrc` 中是否重复执行 `atuin init`、`fzf --zsh`、`zoxide init` 或 Starship 初始化。

常见症状如下：

| 症状 | 最可能的边界 | 优先动作 |
| --- | --- | --- |
| Ghostty 无法创建窗口 | Ghostty 配置或应用版本 | 使用备用终端，恢复 `config.ghostty` |
| `zsh -f` 正常，新 Shell 失败 | Zsh 启动文件 | 语法检查并恢复最近配置 |
| 只有补全或高亮失败 | Antidote bundle 与加载顺序 | 检查静态文件是否刷新、高亮是否最后加载 |
| `Ctrl-R` 行为错误 | fzf 与 Atuin 重复绑定 | fzf 加载时禁用其 `Ctrl-R`，随后只初始化一次 Atuin |
| Atuin 不记录 IDE 命令 | IDE 不是交互式 Shell | 在该终端运行 `atuin doctor`，检查 `shell.preexec` |
| 进入大型仓库后提示符变慢 | Starship 模块或外部命令 | 运行 `starship timings`，缩小耗时模块 |
| SSH 后提示 unknown terminal | 远端缺少 Ghostty terminfo | 按 [[Ghostty 常用配置与 Shell 集成]] 处理，不在服务器安装 Ghostty GUI |

## 7. 完整回退

### 回退配置

从 dotfiles 的已知良好提交或更新前备份恢复配置，然后先检查语法，再开新 Shell：

~~~bash
zsh_config_dir="$(zsh -c 'print -r -- "${ZDOTDIR:-$HOME}"')"
zsh -n "$zsh_config_dir/.zshrc"
exec zsh -l
~~~

若 `.zshrc` 已无法启动，先进入 `zsh -f`，从其中编辑或复制配置。不要关闭最后一个可用终端。

### 回退 Antidote 插件

~~~zsh
antidote snapshot list
snapshot_file="${ZDOTDIR:-$HOME}/snapshots/antidote.txt"
antidote snapshot restore "$snapshot_file"
~~~

路径必须替换为真实 snapshot；恢复后新开终端验证。若插件声明本身也发生变化，还要把 `.zsh_plugins.txt` 恢复到同一配置版本，否则下次刷新可能再次改变插件集合。

### 回退二进制

用原包管理器提供的版本固定、降级或回滚机制恢复单个组件，并保持用户配置不变。不要为了回退 Starship、fzf 或 zoxide 而删除整个 Zsh 配置，也不要为了回退 Ghostty 而修改远程服务器。

### 回退 Atuin 数据

Atuin 数据恢复与普通配置不同：先注释 `atuin init`、关闭其他交互式 Shell，再从受保护的私有备份恢复实际数据目录，最后重新启用集成并运行 `atuin doctor`。同步用户还必须拥有原加密密钥；密钥丢失不能靠重新登录或 Git 仓库恢复。

## 8. 在新机器恢复

按依赖从底层到上层执行：

1. 桌面机器安装 Ghostty；纯服务器或 SSH 目标只需要 Zsh 和 CLI 工具。
2. 按平台安装 Zsh、Antidote、Starship、Atuin、zoxide 与 fzf 二进制。
3. 检出 dotfiles，放置 `.zshenv`、`ZDOTDIR` 配置、插件清单、Starship、Ghostty 和审查后的 Atuin 配置。
4. 新开 Zsh，让 `antidote load` 克隆声明的 bundle；需要精确版本时，再运行 `antidote snapshot restore <已跟踪快照>`。
5. Atuin local-first 用户从空本地数据库开始，或从私有备份恢复；需要同步时，用密码管理器中的密钥登录并执行 `atuin sync`。
6. 不复制 Antidote 缓存、`.zsh_plugins.zsh`、`.zcompdump`、Starship 缓存或 zoxide 数据库来“加速恢复”。
7. 完整执行第 5 节验证，并确认真实 SSH 场景后再把旧机器视为可下线。

这个流程把 Git 中的声明当作配置来源，把 Antidote snapshot 当作插件版本来源，把 Atuin 私有备份或可选同步当作历史数据来源；三者不能互相替代。
