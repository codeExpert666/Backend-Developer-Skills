`nc` 又称 Netcat，可以主动建立 TCP 连接、发送 UDP 数据，也可以进入监听模式。本篇只使用它从实际客户端探测一个已经确认、已有权访问的 TCP 地址与端口，不展开监听、转发、文件传输或批量端口扫描。

客户端使用 `nc` 回答“我现在能否与这个地址和端口建立 TCP 连接”；服务端使用 `ss` 回答“本机现在是否存在对应监听套接字”。二者观察位置不同，应该配合而不是互相替代。

服务端监听状态见 [[Linux 端口、监听套接字与 ss 命令基础]]；地址、路由和 DNS 见 [[Linux 网络接口、IP 地址、路由与 DNS 基础]]；主机防火墙见 [[Linux 主机防火墙与 UFW 基础]]；SSH 的协议与身份验证见 [[OpenSSH 连接、密钥与主机指纹]]。

> [!abstract] 本篇掌握目标
> - **必须熟练**：能够拆解 `nc -vz -w 5 "$TARGET_HOST" "$TARGET_PORT"`，并解释成功只证明当前 TCP 连接成立。
> - **理解会查**：能够根据成功、明确拒绝、超时和名称解析失败选择下一层检查，并使用本机帮助确认选项。
> - **认识即可**：UDP 探测、监听模式、代理、端口范围和不同 Netcat 实现的低频选项；有明确需求时再深入。

> [!info] 核对日期与实现差异
> 本文于 **2026-07-20** 核对 OpenBSD `nc(1)`、Ubuntu 24.04 `netcat-openbsd` 手册和当前 macOS 自带的 `/usr/bin/nc`。本文主线中的 `-v`、`-z` 与 `-w` 适用于这些实现；其他系统中的 `nc` 可能是传统 Netcat 或 Nmap Ncat，执行前应以该客户端上的 `nc -h` 和 `man nc` 为准。

## 1. 先区分服务端观察与客户端验证

| 观察位置 | 要回答的问题 | 常用工具 |
| --- | --- | --- |
| 服务端本机 | 内核当前是否存在监听套接字，绑定在哪个地址和端口 | `ss -lntp` |
| 主机防火墙 | 到达本机的 TCP 连接是否被规则允许 | `ufw status`、防火墙规则 |
| 实际客户端 | 从这台客户端到目标地址和端口能否建立 TCP 连接 | `nc -vz` |
| 应用协议 | 对端是否真在运行预期协议，协议交互是否正常 | `ssh`、`curl` 等协议客户端 |
| 身份认证 | 主机身份、用户或应用凭据是否正确 | SSH 指纹与密钥、TLS 证书、应用账号 |

服务端出现监听行，不代表客户端路径上的路由和防火墙都允许连接；客户端 TCP 探测成功，也不代表对端一定运行预期应用，更不代表应用身份认证已经成功。

## 2. 一次 TCP 端口探测经过哪些层次

```text
目标名称解析为地址
  -> 客户端选择路由和源地址
  -> 中间网络与访问控制允许数据包通过
  -> 目标地址存在对应 TCP 监听入口
  -> TCP 连接建立或返回失败
```

`nc` 观察的是这些层次组合后的结果。它不会直接告诉你失败发生在哪个设备或哪条规则，因此输出是下一步排查线索，而不是根因证明。

如果目标使用主机名，测试还依赖客户端当时的名称解析结果。成功结论只适用于这次解析得到的地址、这台客户端、这个目标端口和当前时间，不能自动推广到其他客户端或未来状态。

## 3. 先确认当前客户端使用哪一种 `nc`

在实际发起连接的客户端上查询：

```bash
command -V nc
nc -h 2>&1 | sed -n '1,120p'
man nc
```

`command -V` 显示当前 Shell 最终会执行哪个程序；`nc -h` 用于快速确认选项；`man nc` 提供该实现的完整说明。帮助信息可能写入标准错误，所以示例使用 `2>&1` 后再交给 `sed` 截取前 120 行。

macOS 通常已经提供 `/usr/bin/nc`。Ubuntu 若提示 `nc: command not found`，先确认客户端确实需要该工具；在 Ubuntu 上可安装提供本文主线语法的 `netcat-openbsd`：

**风险：系统变化；仅在 Ubuntu 客户端缺少 `nc` 时执行。**

```bash
sudo apt update
sudo apt install netcat-openbsd
```

安装完成后重新执行 `command -V nc` 与 `man nc`，不要只凭软件包安装成功就假定命令行为与其他平台完全相同。

## 4. 拆解 `nc -vz -w 5 "$TARGET_HOST" "$TARGET_PORT"`

下面是命令骨架，其中两个变量必须已经保存经过核对的目标名称或地址与端口：

```bash
nc -vz -w 5 "$TARGET_HOST" "$TARGET_PORT"
```

| 部分 | 含义 | 在本命令中的作用 |
| --- | --- | --- |
| `nc` | Netcat 客户端程序 | 尝试连接目标地址和端口 |
| `-v` | verbose | 输出更详细的成功或失败信息 |
| `-z` | zero-I/O | 只探测监听入口，不向对端发送应用数据 |
| `-w 5` | 等待时间上限 | 对本文核对的实现，将无法建立或空闲连接的等待限制为 5 秒 |
| `"$TARGET_HOST"` | 目标参数 | 可解析名称或数字地址；双引号防止 Shell 继续拆分变量内容 |
| `"$TARGET_PORT"` | 端口参数 | 已确认的单个 TCP 目标端口 |

组合短选项 `-vz` 也可以写成 `-v -z`。因为没有使用 `-u`，本文核对的实现默认使用 TCP。

`-z` 不发送 SSH、HTTP 等应用层请求，但仍会实际发起 TCP 连接尝试，可能出现在服务端日志或安全监控中。它不是“不产生网络流量”的本地查询。

> [!warning] 只测试有权访问的单个目标
> `nc` 也支持端口范围和监听模式，但这些能力不属于本文主线。不要将单端口故障验证扩展为对不属于自己的主机或网段进行批量扫描。

## 5. 标准输出、标准错误与退出状态

OpenBSD 系 `nc` 会把详细诊断信息写入标准错误。因此，直接在终端执行时能够看到消息，但只使用 `>` 重定向标准输出可能得不到这些信息。

需要同时保存普通输出与诊断信息时，可以显式合并：

```bash
nc -vz -w 5 "$TARGET_HOST" "$TARGET_PORT" >nc-check.log 2>&1
```

这会在当前目录创建或覆盖 `nc-check.log`，不应在用途未知的目录照抄。临时排查通常直接查看终端输出即可，无需保存包含内部主机名或地址的日志。

对单个目标端口，本文核对的实现会用退出状态表示这次探测是否成功。脚本中应判断退出状态，而不是依赖某一句可能随实现或语言环境变化的英文输出：

```bash
if nc -vz -w 5 "$TARGET_HOST" "$TARGET_PORT"; then
  printf '%s\n' 'TCP 连接探测成功。'
else
  printf '%s\n' 'TCP 连接探测失败，需要继续分层检查。' >&2
fi
```

这段骨架只负责判断 `nc` 的成功或失败；变量赋值、输入校验和具体排查仍应由调用场景负责。

## 6. 怎样解释常见结果

实际文字会随操作系统、`nc` 实现和语言环境变化，应先识别结果类别：

| 结果类别 | 能说明什么 | 不能直接证明什么 |
| --- | --- | --- |
| 连接成功 | 从当前客户端到本次解析得到的目标地址和 TCP 端口完成了连接 | 对端一定是预期应用、身份可信、认证会成功 |
| 明确拒绝 | 目标路径返回了明确拒绝；常见于没有监听入口、绑定地址不匹配或某层主动拒绝 | 一定是服务未启动，或一定是主机防火墙 |
| 超时 | 在限定时间内没有完成连接 | 具体是地址、路由、丢弃型防火墙、上游策略还是目标离线 |
| 名称解析失败 | 客户端不能把当前名称转换为可用地址 | 使用数字地址时的路由和端口一定有问题 |
| `command not found` | 当前 Shell 找不到 `nc` | 网络或目标服务本身失败 |

一次失败现象可能有多个原因。例如丢弃型防火墙通常表现为超时，明确拒绝型规则可能表现为立即拒绝；中间设备也可以生成拒绝，因此不能仅凭客户端一句输出锁定服务端根因。

## 7. 成功的 TCP 探测没有验证应用协议

`nc -vz` 只尝试建立 TCP 连接。即使测试 TCP 22 成功，也不能证明：

- 对端程序一定是 OpenSSH，而不是绑定在该端口的其他程序。
- SSH 主机公钥与预期指纹一致。
- 登录用户名存在。
- 用户私钥、`authorized_keys` 或目录权限正确。
- 后续新的 TCP 连接一定持续成功。

验证 SSH 时还必须使用 `ssh` 完成协议握手、主机指纹核对和用户认证。验证 HTTPS 时应使用能够处理 HTTP 与 TLS 的 `curl` 等客户端。应选择与目标应用协议匹配的工具，而不是把“TCP 可连接”写成“应用已经正常”。

## 8. SSH 场景中的分层顺序

服务端先核对有效配置与监听：

```bash
sudo sshd -T | grep '^port '
sudo ss -lntp
sudo ufw app info OpenSSH
```

实际客户端再核对 TCP 路径：

```bash
nc -vz -w 5 "$SSH_HOST" 22
```

端口 22 只是新安装 OpenSSH 的常见默认值。如果 `sshd -T` 显示其他有效端口，应将同一个可信端口用于服务端监听检查、UFW 规则、`nc` 和 `ssh -p`，不能只改其中一处。

TCP 探测成功后，再执行 `ssh` 并独立核对主机指纹。完整顺序见 [[OpenSSH 连接、密钥与主机指纹#4. 取得地址并验证端口]]。

## 9. 与其他网络工具的边界

| 工具 | 主要回答的问题 |
| --- | --- |
| `ip` | 本机接口、地址和路由当前是什么 |
| `ss` | 服务端本机有哪些套接字正在监听或已经连接 |
| `ping` | ICMP 探测是否得到响应；失败不等于 TCP 一定不可用 |
| `nc -vz` | 当前客户端能否与目标 TCP 地址和端口建立连接 |
| `curl` | HTTP、HTTPS、TLS 和响应层是否工作 |
| `ssh` | SSH 协议、主机身份和用户认证是否工作 |

排查时按实际问题选择工具，不要用一个工具的成功替代其他层的验证。

## 10. 常见误判

### TCP 22 成功，所以 SSH 密钥没有问题

不成立。密钥验证发生在 TCP 连接建立后的 SSH 协议阶段。

### `Connection refused` 就是 UFW 拒绝

不成立。没有监听套接字、监听地址不匹配、主机或中间设备主动拒绝都可能产生相似现象。

### 超时就应该延长到几十分钟

不成立。较短的等待时间用于避免一次明显失败长期阻塞；超时后应转向地址、路由、监听和访问控制检查，而不是只不断延长等待。

### `-z` 不发送应用数据，所以执行完全没有外部影响

不成立。它仍会发起网络连接，可能触发服务端日志、连接计数或安全监控。

### 所有名为 `nc` 的命令选项都相同

不成立。先识别实际可执行文件和实现，再以当前客户端的帮助与手册为准。
