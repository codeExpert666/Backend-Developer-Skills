端口是 TCP、UDP 等传输层协议在一台主机上区分通信入口的数字标识；套接字是 Linux 内核维护的通信端点，包含协议、本地地址与端口、状态等信息。服务只有创建并监听了合适的套接字，内核才有入口把新到达的连接交给它。

`ss` 用于读取 Linux 当前的套接字状态。它能回答“现在是否存在监听套接字、绑定在哪个本地地址和端口、由哪个进程持有”，但不会启动服务、修改服务配置或放行防火墙。

接口、地址和路由见 [[Linux 网络接口、IP 地址、路由与 DNS 基础]]；从实际客户端测试 TCP 路径见 [[TCP 端口连通性测试与 nc 命令基础]]；主机入站规则见 [[Linux 主机防火墙与 UFW 基础]]；SSH 场景见 [[OpenSSH 连接、密钥与主机指纹]]。命令结构和帮助系统见 [[Shell 命令结构、类型与帮助系统]]。

> [!abstract] 本篇掌握目标
> - **必须熟练**：区分端口、监听套接字、进程和防火墙，能够拆解并阅读 `sudo ss -lntp` 的基础输出。
> - **理解会查**：根据 TCP、UDP、监听或已建立连接选择选项，忘记过滤语法时会使用 `ss --help` 和 `man ss`。
> - **认识即可**：队列细节、TCP 内部状态、网络命名空间和复杂过滤表达式；遇到对应问题时再深入。

> [!info] 核对日期与适用范围
> 本文于 **2026-07-20** 核对 iproute2 的 `ss(8)` 手册。不同 iproute2 版本、终端宽度和当前用户权限可能改变输出布局或可见的进程信息，应以目标 Linux 主机上的 `man ss` 与实际输出为准。

## 1. 先区分端口、套接字与服务进程

下面几个对象处于同一条连接路径中，但不是同一回事：

| 对象 | 负责什么 | 示例 |
| --- | --- | --- |
| 协议 | 定义通信方式 | TCP、UDP |
| 本地地址 | 决定套接字绑定到本机哪些地址 | `127.0.0.1`、`0.0.0.0`、某个接口地址 |
| 端口 | 在主机和协议范围内标识通信入口 | SSH 默认通常使用 TCP 22 |
| 套接字 | 由内核保存协议、地址、端口和状态 | TCP 监听套接字、已建立连接 |
| 服务进程 | 创建或接管套接字并处理数据 | `sshd`、Web 服务器、数据库 |

“配置文件写了端口 22”只表示期望配置；“存在监听套接字”才表示内核当前确实有接收入口。使用 systemd socket activation 时，也可能先由 systemd 持有监听套接字，连接到达后再激活真正的服务进程。

TCP 会先建立连接，因此服务端通常出现 `LISTEN` 状态。UDP 没有同样的连接建立过程，已绑定并等待数据的 UDP 套接字可能显示为 `UNCONN`，不能要求它也显示 `LISTEN`。

## 2. 监听只是连接链路中的一层

从客户端连接一个 Linux 服务，至少需要依次核对：

```text
服务期望配置
  -> 当前监听套接字
  -> 主机防火墙与上游访问控制
  -> 客户端到目标地址和端口的实际连接
  -> 应用协议与身份认证
```

因此：

- 服务配置正确，不代表服务已经启动或套接字已经监听。
- 套接字正在监听，不代表 UFW、云安全组或上游网络允许连接。
- 防火墙允许端口，不代表真的有服务监听。
- TCP 连接成功，不代表 SSH 密钥、数据库账号或应用认证一定成功。

“端口已打开”容易把多个层次混在一起。排查时应改为说明具体事实，例如“TCP 22 正在 `0.0.0.0` 上监听”或“UFW 允许某个来源访问 TCP 22”。

## 3. 拆解 `sudo ss -lntp`

```bash
sudo ss -lntp
```

组合短选项也可以写成 `sudo ss -l -n -t -p`。每一部分含义如下：

| 部分 | 含义 | 在这条命令中的作用 |
| --- | --- | --- |
| `sudo` | 以管理员权限运行后面的命令 | 尽量完整显示其他用户所拥有套接字的进程信息；不是 `ss` 的选项 |
| `ss` | 查看套接字统计和状态 | 从当前 Linux 内核读取套接字信息 |
| `-l` | `--listening` | 只显示监听套接字 |
| `-n` | `--numeric` | 保留数字形式，不把端口转换成 `ssh` 等服务名 |
| `-t` | `--tcp` | 只查看 TCP 套接字 |
| `-p` | `--processes` | 显示使用套接字的进程信息 |

`sudo ss -lntp` 是只读查询，`sudo` 在这里不代表会修改系统。但不能由此推断整个 `ss` 命令族都只读；例如低频的 `ss -K` 会尝试强制关闭匹配的套接字，不应在只想观察状态时照抄。

如果不需要进程信息，可以先使用不带 `sudo` 的最小骨架：

```bash
ss -lnt
```

## 4. 阅读输出时先看哪些列

不同版本和终端宽度可能让 `Process` 信息换行。下面只是字段形态示例，不是某台主机的预期事实：

```text
State  Recv-Q Send-Q Local Address:Port Peer Address:Port Process
LISTEN 0      4096   0.0.0.0:22       0.0.0.0:*       users:(("sshd",pid=1234,fd=3))
```

第一阶段按以下顺序阅读：

| 字段 | 先回答什么 |
| --- | --- |
| `State` | TCP 套接字是否处于 `LISTEN` |
| `Local Address:Port` | 服务当前绑定在哪个本地地址和端口 |
| `Process` | 套接字由哪个进程持有；权限不足时可能缺失 |
| `Peer Address:Port` | 对监听套接字通常是通配值，不表示某个客户端已经连接 |
| `Recv-Q`、`Send-Q` | 队列相关数值；含义会随套接字状态变化，不是本阶段判断“是否监听”的首要字段 |

不要只在整行中搜索进程名。使用 systemd socket activation 时，`Process` 可能显示 `systemd` 而不是最终处理连接的服务；还可能因为权限限制而不显示完整信息。应优先比较实际协议、`State`、`Local Address:Port` 和服务管理器状态。

## 5. 监听地址决定哪些本地入口可用

同一个端口绑定到不同地址，可达范围不同：

| `Local Address` 形态 | 基础含义 |
| --- | --- |
| `127.0.0.1:8080` | 只绑定 IPv4 回环地址，其他主机不能直接连接这个入口 |
| `[::1]:8080` | 只绑定 IPv6 回环地址 |
| `0.0.0.0:22` | 绑定所有本地 IPv4 地址 |
| `[::]:22` | 绑定所有本地 IPv6 地址；是否同时接收 IPv4 取决于系统和套接字设置，不能只凭这一行假定 |
| `192.0.2.10:22` | 只绑定示例中的某个本地地址；实际值必须属于当前主机 |
| `*:22` | 通配显示形式，需结合协议族和当前输出判断具体范围 |

绑定所有本地地址也不等于互联网一定能够访问。客户端仍需使用可达地址，中间路由、虚拟化网络、主机防火墙和上游访问控制也必须允许连接。

## 6. 按问题选择常用骨架

### 查看 TCP 监听套接字

```bash
ss -lnt
```

### 同时查看持有 TCP 监听套接字的进程

```bash
sudo ss -lntp
```

### 查看 UDP 监听或已绑定套接字及进程

```bash
sudo ss -lnup
```

UDP 行的状态可能显示为 `UNCONN`。这里的 `-u` 选择 UDP，不能与 `-t` 的 TCP 含义混用。

### 查看已经建立的 TCP 连接

```bash
ss -tn state established
```

这回答的是“哪些 TCP 连接已经建立”，与 `-l` 查询监听入口不是同一个问题。

### 只查看已经确认的 TCP 端口

如果已经从可信配置确认目标端口确实是 22，可以让 `ss` 自己按本地端口过滤：

```bash
sudo ss -lntp 'sport = :22'
```

`sport` 在这里匹配本地源端口。目标服务使用其他端口时，应先从它的有效配置取得真实值，而不是为了得到输出就猜一个端口。复杂过滤表达式用到时再查 `man ss`，不必提前背诵。

## 7. 在 SSH 排查中怎样使用

检查 SSH 时应比较三个不同事实：

```bash
sudo sshd -T | grep '^port '
sudo ss -lntp
sudo ufw app info OpenSSH
```

1. `sshd -T` 读取 OpenSSH 当前解析后的有效配置。
2. `ss` 读取 Linux 内核当前实际存在的 TCP 监听套接字。
3. `ufw app info OpenSSH` 读取 UFW application profile 声明的协议和端口。

三者的端口一致，仍只完成了服务端检查。最终还需要从真实客户端建立新连接，并继续完成 SSH 主机身份和用户身份验证，详见 [[OpenSSH 连接、密钥与主机指纹#3. 在服务端准备 sshd]] 与 [[Linux 主机防火墙与 UFW 基础#6. 先比较 SSH 配置、监听状态与 UFW profile]]。

> [!warning] 不要只用 `grep sshd` 判断 SSH 是否监听
> Ubuntu 可以由 `ssh.socket` 先持有监听套接字，进程信息可能显示为 `systemd`。先用有效配置确认端口，再比较 `Local Address:Port` 与 `systemctl status ssh.socket ssh.service`，比只执行 `ss -lntp | grep sshd` 更可靠。

## 8. 常见误判

### 有监听行，所以远程一定能连接

不成立。还要检查绑定地址、客户端使用的目标地址、路由、虚拟化网络、UFW 或其他访问控制。

### 没有进程信息，所以没有程序监听

不一定。先看是否存在对应协议、状态、地址和端口；普通用户可能看不到其他用户的完整进程信息，再根据需要使用 `sudo` 复查。

### `Connection refused` 一定是防火墙拒绝

不一定。它常见于目标地址可达但对应端口没有监听入口，也可能来自中间设备的明确拒绝。应在服务端检查监听和绑定地址，再结合客户端现象与防火墙规则判断。

### 超时一定是服务没有启动

不一定。超时更常把排查方向指向地址、路由、防火墙或上游策略；仍需用 `ss` 独立确认服务端是否监听。

### 宿主机上的 `ss` 能看到所有容器套接字

不一定。容器可以位于独立网络命名空间，宿主机查询、容器内查询和宿主端口发布是不同观察位置。容器端口问题应同时检查容器内部监听、端口发布和主机访问控制。

## 9. 查询帮助与完成检查

忘记选项或过滤语法时，在目标 Linux 主机查询：

```bash
ss --help
man ss
```

完成本文后，应能够：

- [ ] 解释端口、监听套接字、进程与防火墙的区别。
- [ ] 拆解 `sudo ss -lntp` 中的命令和每个选项。
- [ ] 从 `State` 和 `Local Address:Port` 判断 TCP 服务当前监听在哪里。
- [ ] 解释为什么监听行不能证明客户端一定能够连接。
- [ ] 解释为什么 systemd socket activation 场景不应只搜索 `sshd` 进程名。
